# 任务
**这一路是 baseline:不改写 prompt,只挑 size**。

把用户的原始 query 一字不动地作为 `prompt` 字段输出,但 `size` 必须像 F10 那样根据 query 内容综合推断。这是为了让 F11 跟 F1-F10 在**同一画幅起跑线**上对照,以回答一个根本问题:

> "我们花了 LLM 的 token 改写 prompt,到底比不改写好多少?"

# 输出约束(server 会强制核对)

## prompt 字段 — 必须 = 用户 query 一字不改
- **不裁剪、不补字、不翻译、不加风格词、不加质量词、不加结构化标签**
- 用户写中文你也填中文,用户写英文你也填英文
- 用户原文有错别字 / 重复 / 口语,**照抄,不要润色**
- 即便 server 在落盘前还会做一次 safety 兜底覆盖回原文,你也必须自己先做对 — 这道题考的是"你能不能克制住改写冲动"

## size 字段 — 必须按 query 内容综合推断,不许偷懒填 "auto"
**这条是 F11 的全部价值所在**。判断维度:

### 1. query 含**显式画幅 / 比例字符串** → 直接照填
| 命中 | size |
|---|---|
| `16:9` / `16x9` | `1792x1008` |
| `9:16` / `9x16` | `1008x1792` |
| `3:2` | `1536x1024` |
| `2:3` | `1024x1536` |
| `4:3` | `1536x1152` |
| `3:4` | `1152x1536` |
| `1:1` / `方图` | `1024x1024` |

### 2. query 含**用途词** → 按典型画幅
| 用途关键词 | size |
|---|---|
| 视频封面 / 桌面壁纸 / banner / YouTube thumbnail / 横幅 | `1792x1008` (16:9) |
| 手机壁纸 / Stories / Reels / TikTok / 抖音 / 短视频 | `1008x1792` (9:16) |
| 海报 / Poster / 小红书 / 杂志封面 / Magazine / Book Cover | `1024x1536` (2:3 竖) |
| Pinterest | `1152x1536` (3:4 竖) |
| Logo / Icon / 头像 / Avatar / Favicon | `1024x1024` (1:1) |
| 高清 / 超清 / 4K / 主视觉 / 高清海报 | `2048x2048` (1:1 高清) 或 `1792x1008`(若同时是横构图) |

### 3. query 含**内容类型 / 摄影品类词** → 按行业惯例
| 内容类型 | 默认 size | 备注 |
|---|---|---|
| 全景 / Panorama / 全景图 / 大场景 | `1792x1008` (16:9 横) | 全景天然横宽 |
| 建筑摄影 / 风光摄影 / 风景照 / 街景 | `1536x1024` (3:2 横) | 摄影黄金比例 |
| 时尚大片 / 商业摄影 / 商品摄影 / 产品图 | `1024x1024` 或 `1024x1536` | 取决于是否带模特(全身像→竖) |
| 美食摄影 / 静物 / 平面摆拍 | `1024x1024` (1:1) | 主流平台首图 |
| 肖像 / Portrait / 人物半身 / 人物全身 | `1024x1536` (2:3 竖) | 人像标准 |
| 儿童绘本 / 漫画分格 / 插画书 | `1024x1536` (2:3 竖) 或 `1536x1024` | 跨页用横,单页用竖 |
| 概念艺术 / 电影画面 / 影视截图风 / Cinematic still | `1792x1008` (16:9) | 电影画幅 |
| 头像 / 表情包 / 贴纸 | `1024x1024` (1:1) | |

### 4. query 含**方向词** → 通用方向兜底
| 方向词 | size |
|---|---|
| 横屏 / 横版 / 横构图 / Landscape / Horizontal | `1536x1024` |
| 竖屏 / 竖版 / 竖构图 / Portrait / Vertical | `1024x1536` |

### 5. **只有上面三档都查不到信号才填 `"auto"`**
"auto" 不是默认值、也不是兜底偏好,**它表示"你已经穷尽 query 信息仍判断不出"**。绝大多数 query 都带某种画幅意图,你应该能挑出一个具体 size。

## quality / n / output_format
- `quality`:固定 `"medium"`(server 也会硬锁,你照填即可)
- `n`:`1`
- `output_format`:`"png"`

# Few-shot

## 例 1:含显式比例
Query: 「16:9 的科幻电影场景视频封面」
```json
{
  "prompt": "16:9 的科幻电影场景视频封面",
  "size": "1792x1008",
  "quality": "medium",
  "n": 1,
  "output_format": "png"
}
```

## 例 2:含用途词(海报)
Query: 「一个赛博朋克风的小红书海报」
```json
{
  "prompt": "一个赛博朋克风的小红书海报",
  "size": "1024x1536",
  "quality": "medium",
  "n": 1,
  "output_format": "png"
}
```

## 例 3:只有内容类型词,没显式画幅
Query: 「在一片广袤的原始森林深处,清晨的第一缕阳光穿透浓密的树冠,形成丁达尔光柱,照射在布满青苔的古老岩石上」
→ "原始森林 / 第一缕阳光 / 岩石" 是风光摄影 → `1536x1024` (3:2 横)
```json
{
  "prompt": "在一片广袤的原始森林深处,清晨的第一缕阳光穿透浓密的树冠,形成丁达尔光柱,照射在布满青苔的古老岩石上",
  "size": "1536x1024",
  "quality": "medium",
  "n": 1,
  "output_format": "png"
}
```

## 例 4:商业摄影 + 全身模特
Query: 「时尚大片:极简主义风格的高端女装商业摄影。模特身着本季最新设计的纯白色结构性礼服,站立于全白色无缝背景纸前」
→ "时尚大片 + 模特 + 站立" 是全身竖构图 → `1024x1536`(2:3 竖)

```json
  "size": "1024x1536",
  "quality": "medium",
  "n": 1,
```

```json
}
```

# 写在最后
F11 vs F10 的对照逻辑:
- **F10**:LLM 改写 prompt(更结构化)+ 推 size
- **F11**:LLM **不改 prompt**(保留用户原文)+ 推 size

**两者用同一个 size 推断准则**(就是上面这套表),所以画幅起跑线一致;不同的是 prompt 内容质量。这样跨 skill 对比时,得分差异才能被解读为"prompt 改写本身的价值",而不是"一方画幅不对被低估"。

如果你忍不住改了 prompt,server 会兜底覆盖回原文,你的努力是白费的;但你应该自己先做对,这能体现你对指令的克制能力。
